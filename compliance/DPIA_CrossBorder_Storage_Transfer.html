1. Introduction and Purpose

This Data Protection Impact Assessment (DPIA) has been prepared in accordance with Law No 058/2021 of 13/10/2021 relating to the protection of personal data and privacy in Rwanda. It assesses the privacy risks associated with the cross-border storage and transfer of personal data for the Bara Afrika digital platform and the measures taken to mitigate those risks.

This document is submitted as a supporting attachment for:

  • Application for Authorization to Store Personal Data Outside Rwanda
  • Application for Authorization to Transfer Personal Data Outside Rwanda

2. Processing Overview

2.1 Product Description

Bara Afrika is a digital platform that provides the following services to users, businesses, and creators in Rwanda and the region:

  • User account registration and profiles (authentication via Clerk)
  • Business discovery: marketplace listings, reviews, and ratings
  • Events creation, promotion, and ticketing (with ticket purchase confirmations)
  • Music / content streams: song discovery, play history, likes, playlists
  • Blog and community content: creation, submission, approval workflows
  • Admin management: content moderation, listing and event approval, user management

2.4 Special Categories of Sensitive Data

Bara Afrika does not intentionally collect or process special categories of personal data (health, biometric, genetic, racial or ethnic origin, political opinions, religious beliefs, criminal record data, or data concerning a person's sex life). If users incidentally include such information in free-text fields, it is treated as incidental user-generated content and is not sought or used by the platform.

3. Purpose and Lawful Basis

3.1 Purpose of Processing

  • Providing and operating the Bara Afrika platform and its services
  • Authenticating users and managing identity
  • Storing platform data and user-generated content
  • Sending transactional and service notification emails
  • Maintaining security, fraud prevention, and abuse monitoring
  • Administering the platform and moderating content

3.2 Lawful Basis

  • Contract / service performance: Core platform functions (account creation, listing/event submission, ticket purchase) are necessary to perform the service the data subject has subscribed to
  • Legitimate interests: Security monitoring, audit logging, fraud and abuse prevention, service reliability
  • Consent: Only where optional features are involved (e.g., optional marketing communications, where applicable)

4. Necessity and Proportionality

4.1 Necessity of Cross-Border Processing

The cross-border storage and transfer of personal data to these vendors is technically necessary because:

  • Supabase provides the managed database and object storage infrastructure that underpins all platform operations; no equivalent managed service is available locally in Rwanda
  • Clerk provides user authentication and identity management; its EU-based infrastructure offers established security controls and GDPR compliance that reduce risk for data subjects
  • Resend provides transactional email delivery for service notifications (account confirmation, event tickets, listing approvals, etc.) that are functionally necessary for platform operation

4.2 Data Minimization

  • Only data necessary for each service function is collected and shared with each vendor
  • Sensitive personal data categories are not collected
  • Users provide contact details voluntarily (phone is optional)
  • Operational logs are collected only to the extent necessary for security and debugging

4.3 Retention

  • Account data is retained while the account is active
  • Logs are retained for limited periods (security/debugging purposes)
  • Data deletion requests are handled in accordance with applicable law and Bara Afrika's data rights procedure
  • Supabase data is retained as per Bara Afrika's configuration; vendor-side retention is governed by the respective DPA

6. Risk Assessment

6.1 Identified Risks

Risk Severity Likelihood Mitigating Controls Residual Risk
Unauthorized access to personal data at rest (vendor breach) High Low Vendor encryption at rest (AES-256), access controls, vendor security certifications, DPA in place Low
Unauthorized interception of data in transit High Low TLS/HTTPS enforced for all API calls and frontend traffic Low
Unauthorized access via compromised credentials High Low-Medium Secrets management (env vars, not in client code), least privilege, role-based access, key rotation procedures Low-Medium
Excessive data collection / data subject rights violation Medium Low Data minimization, no sensitive categories collected, data rights request procedure in place Low
Vendor subprocessor risk (unauthorized sub-transfer) Medium Low DPAs include sub-processor obligations; sub-processor lists reviewed (Supabase DPA Schedule 3) Low
Data loss / unavailability Medium Low Supabase managed backups and DR, provider SLAs, ability to restore from backup Low
Misconfiguration of access controls (RLS, admin roles) High Low-Medium Supabase Row Level Security (RLS) on user data tables, tiered admin roles (super_admin/admin/moderator), service role key restricted to backend only Low-Medium
Delayed breach notification Medium Low Supabase DPA provides 48-hour incident notification; Breach Response Playbook documented; DPO contact designated Low

7. Measures to Address Risks

7.1 Organizational Measures

  • Designated privacy / security responsible person (DPO contact)
  • Incident response and breach handling procedure: compliance/BREACH_RESPONSE_PLAYBOOK_DRAFT.md
  • Vendor due diligence: DPAs reviewed and maintained for all three vendors
  • Data rights request procedure documented: compliance/DATA_REQUESTS_AND_RIGHTS_DRAFT.md
  • Privacy Policy and Terms of Service drafted and to be published before launch

7.2 Technical Measures

  • TLS / HTTPS for all data in transit (enforced at infrastructure level)
  • Role-based access control: admin roles tiered (super_admin / admin / moderator) controlled by admin_users table
  • Principle of least privilege: service role key used only in server/edge function contexts, not exposed to frontend
  • Supabase Row Level Security (RLS): applied to user data tables where applicable
  • Admin activity audit log: admin_activity_log table records admin management actions
  • Secrets management: API keys and credentials stored as environment variables, not in client-side code or source repository
  • Managed backups via Supabase (see Supabase DPA for backup/DR details)

7.3 Contractual Safeguards

  • Supabase DPA: Data Processing Addendum in place; covers sub-processors, breach notification within 48 hours, data deletion, encryption at rest, access controls, and audit trails. Attached as evidence.
  • Clerk Terms / DPA: Contractual data processing obligations in place for EU-based identity provider. Attached as evidence.
  • Resend Terms / DPA: Contractual data processing obligations in place for email delivery provider. Attached as evidence.
  • Rwanda DPO Standard Contractual Clauses (SCCs) attached where applicable.

8. Residual Risk and Conclusion

Following the application of the technical, organizational, and contractual safeguards described above, the residual risk of the cross-border storage and transfer of personal data is assessed as acceptable. The remaining low-to-medium residual risks are inherent to any cloud-hosted platform and are proportionate to the benefits provided.

Decision: Proceed with cross-border storage and transfer of personal data to Supabase (Switzerland), Clerk (Germany/Ireland), and Resend (United States), subject to obtaining supervisory authorization from NCSA / DPO and maintaining ongoing compliance with Law No 058/2021.

Ongoing Review

  • This DPIA will be reviewed when new vendors are added, data categories change, or a data breach occurs
  • Admin access will be reviewed periodically
  • Privacy documentation will be updated when features or vendors change
