This document summarizes the technical and organizational security measures implemented by Bara Afrika to protect personal data processed on the platform. It is submitted as a supporting attachment for the DPO authorization applications for cross-border storage and transfer of personal data.

1. Identity and Access Management

ControlImplementation Detail
AuthenticationUser authentication is handled by Clerk, an EU-based identity provider (Germany/Ireland). Clerk handles session management, token issuance, and credential security.
Admin access controlAdmin access is restricted to users explicitly listed in the admin_users database table. Access is tiered: super_admin / admin / moderator.
Principle of least privilegeAdmin roles are separated by permission level. The Supabase service role key (full DB access) is restricted to server-side / edge function contexts only — never exposed to the frontend client.
Service account managementAPI keys and secrets are stored as environment variables, not in source code or client-side bundles. The .env file is excluded from version control.

2. Data Storage and Processing

ControlImplementation Detail
Primary data storageSupabase managed PostgreSQL database and object storage. Region: Zurich, Switzerland (eu-central-2). All platform data (user profiles, UGC, events, listings, streams data) is stored here.
Authentication dataManaged by Clerk (Germany / Ireland). Identity data (name, email, phone, session) is held in Clerk's EU infrastructure.
Email deliveryTransactional emails (account notifications, ticket confirmations, listing approvals, etc.) are sent via Resend (United States). Email content is minimal and transactional; no bulk marketing.
Row Level Security (RLS)Supabase RLS policies are applied on user data tables where applicable, ensuring users can only access their own data through the platform API.
Data minimizationOnly data necessary for each feature is collected. Sensitive data categories (health, biometric, political, etc.) are not collected. Phone number is optional.

3. Encryption

ControlImplementation Detail
Data in transitAll external traffic uses HTTPS / TLS. This applies to: frontend-to-API calls, Supabase API calls, Clerk authentication flows, Resend API calls (via Supabase Edge Function).
Data at rest — SupabaseAES-256 encryption at rest for all database data and object storage. Encrypted backups maintained. (Per Supabase DPA, attached.)
Data at rest — ClerkEncryption at rest per Clerk security practices (per Clerk compliance documentation, attached).
Data at rest — ResendEncryption at rest per Resend's infrastructure (per Resend DPA / terms, attached).

4. Logging and Monitoring

ControlImplementation Detail
Admin activity audit logAdmin management actions (adding/removing admins, role changes) are logged to the admin_activity_log table in Supabase. Schema created in migration 20241211_create_admin_users_secure.sql.
Application logsSupabase provides project-level logs for Edge Function invocations, database query activity, and authentication events via the Supabase dashboard.
Email delivery logsResend provides delivery status and event logs for sent emails (accessible via Resend dashboard).
Authentication logsClerk provides authentication event logs (sign-in attempts, session activity) via Clerk dashboard.

5. Backups and Disaster Recovery

ControlImplementation Detail
Database backupsSupabase provides managed automated backups for the database. Backup schedule and retention are governed by the Supabase plan and DPA. Recovery is performed via Supabase's point-in-time recovery capabilities.
DR strategyDisaster recovery relies on Supabase's managed infrastructure. In the event of an incident, restoration is from Supabase backups. No separate DR infrastructure is maintained independently by Bara Afrika at this stage.

6. Incident Response

ControlImplementation Detail
Breach response playbookAn internal breach response process is documented at compliance/BREACH_RESPONSE_PLAYBOOK_DRAFT.md. This covers: identification, containment, investigation, notification, documentation.
Vendor incident notificationSupabase DPA requires security incident notification within 48 hours of becoming aware of a breach affecting customer data. Clerk and Resend provide incident notification per their contractual terms.
DPO notification readinessA designated responsible person (DPO contact) is identified to receive vendor breach notifications and coordinate the Bara Afrika response. The DPO breach report form is referenced: https://dpo.gov.rw/fileadmin/DPO/Services/Personal-Data-Breach-Report-Form.pdf

8. Email Templates Handled (Transactional Only)

The following types of transactional emails are sent via the supabase/functions/send-email Edge Function and Resend. All are operational notifications — no bulk marketing:

  • Welcome email (account registration)
  • Ticket purchased / ticket confirmation
  • Marketplace listing created / approved / rejected
  • Event submitted / approved / rejected
  • Banner request notifications
  • Contact form confirmation
  • Blog submitted / approved / declined
  • Song uploaded / published

9. Ongoing Governance

  • Admin access reviewed periodically (at minimum, quarterly)
  • Vendor registers and privacy documentation updated when features or vendors change
  • This Security Controls Summary to be updated whenever significant changes are made to the security architecture
  • DPO authorization applications to be updated if new cross-border vendors or data categories are added
