4. Grounds for the Transfer

The transfers are requested on the following grounds:

  • Authorization from the supervisory authority — this application seeks that authorization
  • Necessity for the performance of a contract between data subjects and Bara Afrika — to provide the Services the data subjects have subscribed to, the transfers to these vendors are technically necessary
  • Compelling legitimate interests — ensuring security, fraud prevention, service reliability and continuity

We do not rely solely on consent as the grounds for core infrastructure transfers, as the transfers are technically necessary to deliver the service.

5. Safeguards and Protection Measures

5.1 Format and means of transfer

  • Format: Data encrypted in transit (TLS/HTTPS); data at rest encrypted by providers
  • Means of transfer/access: API integration over TLS — the platform backend calls each vendor's API using secure credentials. No bulk data file transfers take place.

5.2 Safeguards implemented by Bara Afrika

  • Role-based access controls and least privilege for all administrative access
  • Admin activity logging (admin_activity_log table in Supabase)
  • Supabase Row Level Security (RLS) on user data tables where applicable
  • TLS/HTTPS on all external traffic
  • Secrets management: credentials stored as environment variables, not in client-side code
  • Vendor DPAs / contracts in place with each recipient (attached)
  • Incident response and breach notification process documented

5.3 Safeguards implemented by recipients

  • Supabase: AES-256 encryption at rest, encrypted backups, least privilege access, audit trails, 48-hour security incident notification (per DPA attached)
  • Clerk: EU GDPR-compliant identity infrastructure, security controls per compliance documentation (attached)
  • Resend: Contractual data processing safeguards per DPA / terms (attached)
